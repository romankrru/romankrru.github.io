# Правила оформления статей

Руководство для markdown-материалов в `src/content/` (блог, work, projects).

Файл не входит в коллекции Astro: это справочник для авторов и агентов.

---

## Пунктуация

### Длинное тире

Между частями фразы — **длинное тире** `—` с пробелами по бокам:

```markdown
PRD живёт только как GitHub issue — в репозитории его нет.
```

Короткий дефис `-` — только внутри составных слов и путей: `issue-трекер`, `prd-<N>`, `ready-for-agent`, `vertical-slice`. Не как знак препинания вместо тире.

### Кавычки

В **теле** русской статьи — кавычки-«ёлочки»: `«раунд»`, `«пресет»`.

В frontmatter (YAML), бэктиках и блоках кода — прямые `"` допустимы.

Английские термины в кавычки не берём (см. ниже).

---

## Буква ё

Пишем **явно** везде, где она произносится: `создаёт`, `берёт`, `всё`, `живёт`, `передаёт`.

---

## Числа

| Случай                                    | Как писать                               |
| ----------------------------------------- | ---------------------------------------- |
| Номера issue/PR, параметры, значения в UI | цифрами: `#13`, `20`, пресеты `10`, `25` |
| Небольшие счётные слова в prose           | словами: «три условия», «три стороны»    |

---

## Английские термины и аббревиатуры

| Тип                              | Оформление                                               | Пример                                                         |
| -------------------------------- | -------------------------------------------------------- | -------------------------------------------------------------- |
| Английские словосочетания        | _курсив_                                                 | _ubiquitous language_, _vertical slice_, _acceptance criteria_ |
| Аббревиатуры                     | без курсива; расшифровка в скобках при первом упоминании | ADR (_Architectural Decision Record_, …), PRD, TDD, AFK, HITL  |
| Буквальные имена в коде, CLI, UI | бэктики                                                  | `CONTEXT.md`, `ready-for-agent`, `gh issue list`               |

Не совмещать курсив и бэктики для одного и того же термина.

---

## Имена файлов, путей, скиллов, команд

Всегда в **бэктиках**, когда это буквальное имя:

- файлы: `CONTEXT.md`, `PROMPT.md`
- пути и ветки: `prd/<N>-<slug>`, `ralph/<n>-<slug>`, `main`
- скиллы: `/grill-with-docs`, `/to-prd`
- лейблы и статусы: `ready-for-agent`, `STATUS=done`
- фрагменты коммит-сообщений: `Closes #<n>`

Ссылки на скиллы в markdown — бэктики внутри текста ссылки:

```markdown
[`/to-prd`](https://example.com/skills/to-prd)
```

**Жирный** (`**…**`) для имён инструментов — только в итоговых чеклистах («Если коротко»), чтобы шаги визуально выделялись. В основном тексте — обычные бэктики без bold.

## Собственные имена

| Сущность                   | Как писать                                 |
| -------------------------- | ------------------------------------------ |
| Цикл / инструмент          | **ralph loop** — строчными, как npm        |
| Персонаж                   | **Ralph Wiggum** — полное имя с заглавными |
| Заголовок раздела про цикл | `### ralph loop`                           |

---

## Опечатки и грамматика

Исправляем в том же проходе, что и типографику (опечатки вроде «C помощью», лишние запятые, «заводится, если»). Отдельные содержательные TODO (пояснить термин, добавить ссылку) в типографику не входят.

---

## Frontmatter

Стандартные поля коллекций — без изменений. Кавычки в YAML — прямые `"`. Поле `lang: "ru"` | `"en"` задаёт язык статьи.

---

## Чеклист перед публикацией

- [ ] Нет `-` вместо `—` в prose
- [ ] Ё везде согласована
- [ ] Русские цитаты в «ёлочках»
- [ ] Файлы, пути, скиллы в бэктиках
- [ ] EN-фразы в _курсиве_, аббревиатуры без курсива
- [ ] ralph loop / Ralph Wiggum по таблице выше
